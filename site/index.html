概要

AICU-bench は、実際の AI 技術のユースケースを想定し、 AI ワークロードのパフォーマンスを多角的に計測する 再現可能なオープンソースベンチマークです。
今回はストレージ速度（PCIe Gen5 NVMe / SATA SSD / HDD）が AI ワークロードに与える影響にフォーカスし、 LLM モデルロード・コード生成・画像/動画生成・音声合成の各タスクで検証。 さらに、長時間負荷での熱影響、モデル頻繁切り替え時の速度安定性、 SSD フラグメンテーションによる性能変化についても評価しています。

同じ M.2 スロットでも速度が違う理由

ノート PC の M.2 スロットは物理的には同じコネクタ (M-Key) ですが、 接続先の PCIe レーン世代と挿す SSD のコントローラの両方で速度が決まります。

デバイスインターフェースカタログ Read実測 Read (本ベンチ)備考
KIOXIA EXCERIA G2
C: OS ドライブ		 PCIe Gen3x4 2,100 MB/s 820–1,052 MB/s Gen3 コントローラ搭載の廉価モデル。OS バックグラウンド I/O の影響も大。
ADATA LEGEND 850
D: データドライブ		 PCIe Gen4x4 5,000 MB/s 1,049–1,340 MB/s Gen4 コントローラ搭載。カタログ値と実測の差は OS やファイルシステムの影響。

ポイント: 同じノート PC の M.2 スロットに挿しても、 Gen3 SSD なら最大 3,500 MB/s、Gen4 SSD なら最大 7,000 MB/s、Gen5 SSD なら最大 14,000 MB/s と、 購入する SSD のコントローラ世代で上限が決まります。 スロット側が Gen4 対応でも Gen3 SSD を挿せば Gen3 速度で動作し、 逆に Gen5 SSD を Gen4 スロットに挿しても Gen4 の上限に制限されます。

さらに、カタログスペックと実効速度には大きな乖離があります（本ベンチでは カタログ値の 20〜40%）。 これは OS のファイルシステムオーバーヘッド、バックグラウンドプロセス、SSD の SLC キャッシュ残量などが影響するためです。 AI ワークロードが実際に体感する速度はカタログ値ではなくこの実効速度です。

ComfyUI 起動パラメータとストレージ I/O の関係

ComfyUI の VRAM 管理方式はモデルのロード・オフロード挙動に大きく影響し、 ストレージ I/O パターンが変化します。本ベンチマークでは以下の 3 条件で計測。

パラメータ挙動ストレージへの影響
--highvram モデルを VRAM に常駐。切り替え時もアンロードしない 初回ロード後はストレージアクセス最小。VRAM 大容量環境向け
デフォルト VRAM 不足時に自動的にモデルをオフロード モデル切り替え時にストレージからの再ロードが発生
--lowvram 積極的に VRAM を解放。処理ステップごとにモデルを移動 ストレージ I/O が最も頻繁。低 VRAM 環境でのストレージ速度の重要性を可視化

96GB VRAM 環境 (RTX PRO 6000) では --highvram で全モデルを常駐可能。 6〜16GB VRAM 環境では --lowvram が必須で、ストレージ速度が体感に直結する。

分析手法: 「秒」だけでは見えないボトルネック

モデルロード時間の単純な秒数比較では、モデルサイズや GPU オーバーヘッドの影響が混在します。
本ベンチマークでは以下の手法でストレージ速度の寄与を分離・正規化して計測します。

📏

sec/GiB 正規化メトリクス

モデルロード時間をモデルサイズ（GiB）で割ることで、 1 GiB あたりのロード秒数を算出。 異なるサイズのモデル間で実効ロード速度を公平に比較できます。

さらに、Ollama の内部計測値 runner_started_s（GPU 初期化時間）を差し引くことで、 純粋なディスク I/O スループット（IO est MB/s）を推定。 Raw ディスクベンチ結果との比率から、デシリアライゼーションや メモリ確保のオーバーヘッドを定量化します。

予備調査では実効ロードスループット（150〜240 MB/s）は Raw ディスク速度（820〜1,340 MB/s）の 1/4〜1/6。
Gen5 NVMe でこの比率がどう変化するかが重要な検証ポイント。
🌡

温度・電力リアルタイムモニタリング

ベンチマーク実行中に nvidia-smi を 10 秒間隔でバックグラウンド記録。 GPU 温度、消費電力、GPU 使用率、VRAM 使用量、クロック周波数、P-State を 時系列 CSV として保存します。

サーマルスロットリング（温度上昇によるクロック制限）が発生するタイミングを 特定し、ベンチマーク結果への影響を評価。 長時間連続負荷でストレージと GPU 双方の温度推移を可視化します。

ラップトップ環境では GPU 温度 80°C 付近でスロットリング開始の傾向。 Samsung 9100 PRO の高度な熱制御がワークステーション環境でどう効くか検証。
🧩

SSD フラグメンテーション影響

NVMe SSD は HDD と異なりシークタイムがないため「フラグメンテーション無関係」とされがちですが、 論理ブロックの断片化は内部的なガベージコレクションや WAF（Write Amplification Factor）に影響 する可能性があります。

新品状態 vs 長期使用（大量のモデルダウンロード・削除を繰り返した状態）で シーケンシャルリード性能に差が出るかを検証。 AI ワークロードは数 GB 単位のファイルを頻繁に書き換えるため、実運用に近い条件です。

Samsung 9100 PRO の 8TB 大容量は空き領域が潤沢で GC 負荷が低く、 長期使用でも性能低下が少ないと予測。
🔄

モデル切り替え & 初回ロードベンチマーク

実際の AI 開発では、用途に応じてモデルを何度も切り替えます。 10〜20 回連続でモデルのアンロード→ロードを繰り返すストレステストに加え、 異なるドライブ間でのモデル切り替え（例: D: → E: → D:）も計測。

初回ロード（コールドスタート）と 2 回目以降（OS キャッシュヒット時）の差、 さらにドライブ切り替え時のキャッシュ無効化による影響を分離します。 GPU 温度モニタリングのデータと照合し、 熱がロード性能に与える影響を定量化します。

初回ロードとキャッシュヒット時で最大 2〜3 倍の差が出ると予測。 ドライブ切り替え時は確実にコールドスタートとなるため、ストレージ速度の差が最も顕著に表れる。 Samsung 9100 PRO の高速読み出しが最も効く場面。

結果を共有

スクリプト終了時に結果が自動送信されます。
手動で送信する場合:

# 結果 JSON を確認
cat results\bench_summary.json

送信されるデータ: CPU/GPU/RAM スペック、ストレージ型番（シリアル番号はハッシュ化）、ベンチマーク結果。個人を特定する情報は含まれません。

プライバシー保護と同一結果の回避

  • ホスト名・SSD シリアル番号は SHA-256 でハッシュ化して送信されます。元の値を復元することはできません。
  • 送信される情報は CPU/GPU/RAM の型番、ストレージの製品名・バスタイプ・ファームウェアバージョン、およびベンチマーク数値のみです。
  • 個人を特定できる情報（ユーザー名、IP アドレス、ファイルパス等）は一切送信されません。
  • SSD の UniqueId（EUI-64）をハッシュ化した device_id により、同一デバイスからの重複送信を検出・統合します。同じ環境で複数回実行しても、最新の結果のみが反映されます。
  • 送信前に results\bench_summary.json の内容を確認できます。送信は任意です（-SkipSubmit フラグで無効化可能）。

予備調査の知見: なぜノート PC では動かなかったか

VRAM の壁: SDXL (6.46GB) は RTX 4050 の 6GB VRAM に収まらず、ページファイルへのスワップが必要。 しかしデフォルトのページファイルサイズでは不足し、OS error 1455 で失敗。 --lowvram モードでも同様の結果。これは 6GB VRAM でのSDXL運用の限界を示すデータ。

ディスク容量の壁: 動画生成モデルは 1 実験あたり 36〜39GB を要求。 1TB NVMe では OS + 開発環境 + AI モデルの共存が困難。 本実験環境の 8TB Samsung 9100 PRO なら全モデルを同時に保持可能で、 モデル切り替え時の再ダウンロードが不要になる。

全実験の合計必要容量: ~97GB（順次実行時の最小必要容量: ~51GB）

発見 1: 理論値は出ない

両ドライブとも NVMe Gen4 接続で、カタログ上のシーケンシャルリードは 3,000〜3,500 MB/s。 しかし実測では最高でも 1,340 MB/s にとどまりました。

理由: CrystalDiskMark 等の専用ベンチマークツールは、OS のファイルシステムを介さず直接 NVMe コマンドで計測します。 本ベンチマークは FileStream（.NET の標準ファイル API）を使うため、 OS のファイルシステムオーバーヘッド、I/O スケジューラ、セキュリティソフトのリアルタイムスキャン等が介在します。 つまり「実際の AI ワークロードが体感する速度」に近い値が得られます。

AI アプリケーション（Ollama, ComfyUI 等）もファイル API 経由でモデルを読むため、 カタログスペックではなくこの「実効速度」がモデルロード時間を決定します。

発見 2: OS ドライブは不利

C: (OS ドライブ) は D: (データドライブ) より一貫して低速。 1024MB Read では C: 888 MB/s vs D: 1,266 MB/s（約 1.4 倍差）。

理由: OS ドライブは Windows Update、検索インデックス (Windows Search)、 アンチウイルス、Superfetch/SysMain、ページファイル I/O 等のバックグラウンド処理が常時走っており、 AI モデルの読み出しと帯域を奪い合います。

AI モデルの保存先は OS ドライブとは別の専用ドライブにすべき。 Samsung 9100 PRO のような大容量 NVMe を AI 専用ドライブとして運用することで、 バックグラウンド I/O の影響を排除し、安定した高スループットを確保できます。

発見 3: 計測値の分散が大きい

同一条件の 3 回計測でも、最大で 2〜3 倍の開き（D: 512MB Read: 379〜1,475 MB/s）が発生。 WriteThrough（OS キャッシュバイパス）や SequentialScan ヒントを使っても、 OS レベルの I/O 競合を完全には排除できません。

対策: 本ベンチマークでは中央値（median）を採用し、外れ値の影響を軽減しています。 本実験環境（RTX PRO 6000 ワークステーション）では、よりクリーンな環境での計測が可能になる見込みです。

ラップトップ環境は熱やバッテリー制御による I/O スロットリングの影響もあり、 デスクトップ / ワークステーション環境の方がストレージ本来の性能を発揮しやすい。

本ベンチマークの位置づけ

disk-speed-bench は CrystalDiskMark のような「SSD 単体の最大性能」を測るツールではなく、 「AI アプリケーションが実際に体感するストレージ速度」を計測するベンチマークです。 OS のファイルシステム、バックグラウンドプロセス、メモリ状況といった現実の条件を含めた実効速度を測定し、 ストレージ選びの実用的な判断材料を提供します。

テストサイズは 256MB / 512MB / 1024MB の 3 段階で実施し、 OS キャッシュの影響度合いやバックグラウンド I/O との干渉を可視化しています。 AI モデルファイルは 1GB〜数十 GB と大きいため、1024MB 以上のテストサイズがより現実に近い指標となります。

予備調査からの仮説

G-Tune P6-I7G50 (RTX 4050 Laptop / 32GB DDR4 / NVMe Gen4 ×2) での予備計測データに基づく仮説。
本実験 (RTX PRO 6000 96GB / 192GB DDR5 / Samsung 9100 PRO Gen5 NVMe) で検証予定。

仮説 1: モデルロードはストレージ I/O バウンド

qwen3:1.7b (1.4GB) のモデルロード時間は D: (KIOXIA) が 7.5秒、C: (ALEG-850, OS ドライブ) が 10.9秒。 OS ドライブはバックグラウンド I/O（Windows Update、検索インデックス、アンチウイルス等）と競合するため、専用ドライブの方が安定して高速です。

解説: LLM のモデルファイルは数 GB〜数十 GB の巨大なバイナリです。 モデルロードとは、このファイルをストレージから GPU メモリ（VRAM）に転送する処理で、 ストレージのシーケンシャルリード速度が直接影響します。 Gen5 NVMe (約 14,800 MB/s) なら 5GB のモデルを理論上 0.3秒 で読み出せますが、 HDD (約 180 MB/s) では 28秒 かかります。

本実験では Gen5 NVMe vs SATA SSD vs HDD で桁違いの差が出ると予測。
Samsung 9100 PRO は Gen5 NVMe の中でもトップクラスの読み出し性能を持ち、 AI モデルの起動時間を最小化できる最適なストレージです。

仮説 2: 推論速度はストレージに依存しない？

コード生成速度は C: が 23.45 tok/s、D: が 19.38 tok/s。 モデルが VRAM にロード済みであれば、推論（トークン生成）は GPU 演算能力とメモリ帯域に依存し、 ストレージ速度は無関係になるはずです。 予備調査でのドライブ間の差（約 17%）は、OS の負荷変動やメモリ帯域の差によるものと推定されます。

解説: 推論（inference）とは、ロード済みのモデルを使って実際にテキストや画像を生成する処理です。 この段階ではデータは既に VRAM 上にあるため、ストレージへのアクセスは発生しません。 tok/s（tokens per second）は 1 秒あたりの生成トークン数で、推論速度の標準的な指標です。

本実験では全ドライブで推論速度がほぼ同一になると予測。
これは逆に、ストレージの影響がモデルロード時に集中することを意味し、 高速ストレージの価値は「待ち時間の短縮」に直結します。 Samsung 9100 PRO を使えば、AI ツールの起動やモデル切り替えのたびに数十秒〜数分の短縮が期待できます。

仮説 3: VRAM 不足時はストレージがスワップのボトルネック

qwen3:8b (5.2GB) は RTX 4050 (6GB VRAM) でギリギリ動作。 VRAM に収まりきらないモデルの一部がシステムメモリや仮想メモリ（ページファイル）に退避される場合、 推論中にもストレージアクセスが発生し、ストレージ速度が推論速度にも影響する可能性があります。

解説: VRAM は GPU 専用の高速メモリです。モデルが VRAM に収まらない場合、 OS はシステムメモリ（RAM）やページファイル（ストレージ上の仮想メモリ）を使って不足分を補います。 この「スワップ」が頻繁に発生すると、ストレージの速度が推論性能のボトルネックになります。 近年の AI モデルは大型化が進んでおり（70B パラメータモデルは 40GB 以上）、 VRAM 不足は多くのユーザーにとって現実的な問題です。

96GB VRAM の本実験環境では発生しにくいが、一般的なゲーミング PC（8〜16GB VRAM）では重要な知見。
VRAM 不足時こそ Samsung 9100 PRO のような高速 NVMe SSD が真価を発揮します。 HDD でスワップが発生すると推論速度が数十分の一に低下する可能性があります。

Samsung 9100 PRO が AI ワークロードに最適な理由

予備調査の結果は、同じ NVMe Gen4 ドライブ間でもモデルロード時間に 最大 1.5 倍の差 が出ることを示しています。 Gen4 同士でこの差が出るのであれば、Gen5 (14,800 MB/s) vs SATA SSD (560 MB/s) vs HDD (180 MB/s) の比較では さらに圧倒的な差が予測されます。

AI ワークフローでは 1 日に何度もモデルの切り替えや再起動が発生します。 画像生成では Checkpoint + LoRA + VAE で 10GB 超、動画生成では 20GB 超のモデルをロードします。 1 回あたり数十秒の短縮が、1 日の作業で数十分〜数時間の時間節約につながります。

Samsung 9100 PRO は PCIe Gen5 の帯域幅をフル活用し、シーケンシャルリード 14,800 MB/s、 8TB の大容量で複数の AI モデルを 1 台に集約可能。 AI クリエイターにとって「待たない AI 環境」を実現する最適なストレージです。
